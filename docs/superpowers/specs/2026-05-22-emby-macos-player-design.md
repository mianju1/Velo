# macOS Emby 桌面播放器设计

日期：2026-05-22

## 背景

当前项目是一个干净的 Tauri 2 + Vue 3 + TypeScript 脚手架，只有少量模板代码。目标产品是一个运行在 macOS 上的桌面端视频与媒体播放器，可以连接外部 Emby 服务器，并作为一个完整的 Emby 客户端使用。

第一版优先实现完整的 Emby 客户端体验，并从一开始使用 mpv 作为播放内核。后续版本再改进更深入的 macOS 原生播放体验、窗口嵌入、硬件加速调优和高级播放器细节。

## 目标

- 支持多个 Emby 服务器和已保存账号。
- 使用本地加密存储保存服务器配置、账号 Token、用户偏好和播放偏好。
- 第一版覆盖 Emby 的主要媒体类型：电影、剧集、音乐、照片和直播电视。
- 从第一版开始使用 mpv 作为播放引擎。
- 优先使用 Direct Play，必要时请求 Emby 转码。
- 将播放开始、进度、暂停、停止和完成状态同步回 Emby。
- 第一阶段面向 macOS 个人使用或小范围分发，不优先处理签名、公证、自动更新和公开 CI。
- 第一版不包含离线下载能力。

## 非目标

- 完整离线媒体库支持。
- macOS App Store 分发。
- 自动更新基础设施。
- 完整的公开开源项目工作流和 CI 要求。
- 第一版不采用 Keychain 优先的凭据存储。
- 第一版不把 mpv 渲染画面嵌入 Tauri WebView。

## 推荐方向

采用「完整客户端骨架 + mpv 播放闭环」路线。

应用需要在 Emby 客户端层、播放会话编排、mpv 控制、本地加密存储和 Vue UI 之间建立清晰边界。这样第一版可以覆盖较完整的 Emby 客户端能力，同时让播放器内核在后续高质量播放体验建设中保持可替换和可扩展。

## 产品范围

### 服务器与账号管理

- 添加、编辑和删除 Emby 服务器。
- 校验服务器 URL，并获取基础系统信息。
- 使用 Emby 用户账号登录。
- 在本地加密存储中保存认证 Token 和账号元数据。
- 启动时展示已保存的服务器和账号。
- 支持在已保存的服务器和用户之间切换。
- Token 失效时引导用户重新登录。

### 媒体库

- 首页包含继续观看、最近添加、收藏和推荐入口。
- 为电影、剧集、音乐、照片和直播电视提供媒体库入口。
- 媒体列表包含加载、空状态和错误状态。
- 详情页根据媒体类型展示海报、背景图、元数据、简介、评分、年份、时长、人物、季、集、专辑、曲目或频道等信息。
- 支持跨媒体类型搜索。
- 在 Emby API 稳定支持的范围内提供基础筛选和排序。
- 第一版图片加载可以直接使用 Emby 图片 URL；本地图片缓存后续再添加。

### 播放

- 支持从媒体详情页、列表、搜索结果和继续观看入口启动播放。
- 使用 Emby `PlaybackInfo` 选择媒体源。
- 优先使用 Direct Play。
- Direct Play 失败或用户设置要求较低码率、兼容输出时，请求转码。
- 通过 Rust 侧播放命令控制 mpv。
- 支持播放、暂停、跳转、音量、静音、全屏、停止、字幕轨选择和音频轨选择。
- 支持电影、剧集单集、音乐曲目、照片预览或幻灯片，以及直播电视流。
- 将播放开始、进度、暂停、停止和完成状态回传给 Emby。

### 设置

- 服务器和账号管理。
- 远程播放默认最大码率。
- 默认字幕语言。
- 默认音频语言。
- 基础播放行为选项。
- 图片缓存和日志设置可以在核心流程跑通后添加。

### 分发

- 面向 macOS 个人使用或小范围使用。
- 支持本地开发和手动 Tauri 构建。
- 文档需要说明 mpv 安装等运行时前置条件。
- 应用签名、公证、DMG 打磨和自动更新延后处理。

## 架构

### 前端结构

```text
src/
  app/
    router/
    stores/
    bootstrap.ts
  features/
    servers/
    libraries/
    media/
    playback/
    settings/
  services/
    emby/
    playback/
    storage/
  shared/
    components/
    types/
    utils/
```

职责划分：

- `app` 负责应用初始化、路由和全局状态注册。
- `features/servers` 负责服务器配置、登录、账号选择和服务器切换。
- `features/libraries` 负责媒体库导航和媒体类型入口。
- `features/media` 负责媒体列表、详情页、搜索和元数据渲染。
- `features/playback` 负责播放 UI 状态和播放器控制。
- `services/emby` 负责非敏感浏览请求的 TypeScript 类型和 API 封装。
- `services/playback` 负责前端播放编排，并调用 Tauri 播放命令。
- `services/storage` 负责前端访问非敏感本地持久化命令。
- `shared` 存放可复用 UI、类型和工具函数。

### Rust/Tauri 结构

```text
src-tauri/src/
  commands/
    server.rs
    playback.rs
    storage.rs
  emby/
    client.rs
    models.rs
  player/
    mpv.rs
    session.rs
  storage/
    encrypted_store.rs
  errors.rs
  lib.rs
```

职责划分：

- `commands/server.rs` 暴露登录、服务器校验、Token 恢复和账号切换命令。
- `commands/playback.rs` 暴露播放启动、暂停、跳转、停止、音量、全屏、字幕和音轨命令。
- `commands/storage.rs` 暴露安全的本地设置操作。
- `emby/client.rs` 处理 Rust 侧需要的敏感或播放相关 Emby 请求。
- `player/mpv.rs` 管理 mpv 进程生命周期和 JSON IPC 通信。
- `player/session.rs` 跟踪当前播放会话和 Emby 上报状态。
- `storage/encrypted_store.rs` 负责本地加密持久化。
- `errors.rs` 定义结构化应用错误。

### 边界规则

- Vue 不能直接持久化敏感 Token。
- 携带 Token 的请求必须通过统一 API 封装。播放相关的 Token 使用应发生在 Rust 侧。
- 前端提交播放意图时只传 `serverId`、`userId`、`itemId` 和可选的 `mediaSourceId`；前端不自行构造原始播放 URL。
- mpv 细节只存在于 `src-tauri/src/player`。
- 本地存储格式后续应可替换，因此 UI 和功能代码必须依赖存储服务 API，而不是依赖具体文件路径。
- 第一版可以使用受应用控制的外部 mpv 播放窗口；原生窗口嵌入延后处理。

## 数据流

### 启动与登录

1. 应用启动后，请求 Rust 读取本地加密配置。
2. Rust 返回已保存的服务器和账号摘要，不暴露不必要的敏感细节。
3. 用户选择已保存的服务器和账号，或添加新服务器。
4. 应用校验 Token，或执行登录。
5. 有效会话进入首页；无效会话返回登录页。

### 浏览

1. 全局状态中保存当前服务器和用户上下文。
2. 媒体页面使用当前上下文调用 Emby API client。
3. 列表、详情页、图片、搜索结果和媒体库页面基于规范化响应类型渲染。
4. 失败状态限制在当前页面或面板内，并提供重试操作。

### 启动播放

1. 用户对某个 Emby 条目选择播放。
2. 前端将 `serverId`、`userId`、`itemId` 和可选媒体源或播放偏好发送给播放服务。
3. Rust 请求 Emby `PlaybackInfo`。
4. Rust 根据媒体源、mpv 能力、用户码率设置和回退规则选择 Direct Play 或转码 URL。
5. Rust 启动 mpv 并打开选定 URL。
6. Rust 向 Emby 发送播放开始上报。

### 播放进度

1. mpv 通过 JSON IPC 发出状态。
2. Rust 将 mpv 事件转换为应用播放状态。
3. Rust 定期向 Emby 上报播放进度。
4. 前端订阅播放状态事件，用于控制栏和状态展示。
5. 同步失败时重试，但不中断播放。

### 停止播放

1. 停止可能来自用户操作、自然播放完成、mpv 失败或应用 / 窗口关闭。
2. Rust 在可行时向 Emby 发送最终停止和进度上报。
3. Rust 释放 mpv 进程资源。
4. 前端刷新相关的继续观看和详情状态。

## 错误模型

从一开始使用结构化错误：

```ts
type AppError = {
  code: string;
  message: string;
  detail?: string;
  recoverable: boolean;
};
```

重要错误类别：

- 服务器 URL 格式错误。
- 网络不可达。
- HTTPS 证书错误。
- Emby 服务器响应不兼容或不支持。
- 登录被拒绝。
- Token 过期。
- 媒体列表或详情加载失败。
- `PlaybackInfo` 失败。
- Direct Play 失败。
- 转码失败。
- 未找到 mpv。
- mpv 启动失败。
- mpv IPC 断开。
- 媒体解码或流播放失败。
- 播放状态同步失败。
- 加密存储读取、写入或解密失败。

存储解密失败时应要求重新登录或重置账号。播放同步失败不应中断播放。Direct Play 失败时，如果服务器支持，应尝试转码。

## 技术选型

### 前端

- 保持 Vue 3 和 TypeScript。
- 添加 Vue Router 管理应用页面。
- 添加 Pinia 管理服务器、用户、媒体库和播放状态。
- 使用轻量 `fetch` 封装处理 Emby API 请求。
- 构建安静、面向桌面端的 UI，强调信息密度、可扫描性、清晰导航和可预测控制。

### 后端

- 保持 Tauri 2。
- 使用 Rust commands 处理敏感存储、播放会话控制和 mpv 管理。
- 第一版使用加密 JSON 存储，因为数据结构会快速变化。
- 当媒体缓存和更丰富的本地索引变得必要时，再考虑 SQLite。
- 将 Keychain 作为后续增强项。

### mpv

- 第一版使用外部 mpv 进程，并通过 JSON IPC 控制。
- 应用需要检测 mpv 是否可用，并给出清晰的安装或配置错误。
- 文档需要说明如何在 macOS 上安装 mpv。
- 后续版本可以评估 libmpv 或原生嵌入，以实现更深度集成。

### 转码

- 从 Direct Play + `PlaybackInfo` 开始。
- 添加用户最大码率设置。
- 添加基础客户端能力声明，包括容器、编码、音频和字幕支持。
- 回退行为需要明确、可解释，并写入日志。
- 电影和剧集播放稳定后，再调优直播电视和字幕策略。

## 主要风险

- 在 macOS 上把 mpv 嵌入 Tauri WebView 较复杂，因此第一版应使用外部 mpv 窗口。
- Emby API 跨所有媒体类型的覆盖面较广，因此第一版中每种媒体类型的深度应保持务实。
- 转码参数和播放能力需要结合真实服务器迭代测试。
- 本地加密存储比明文安全，但如果密钥也在本地，其安全性仍弱于 macOS Keychain。
- 直播电视比点播媒体更容易出问题，主要风险来自流稳定性、延迟和转码。

## 测试策略

- 单元测试 TypeScript API URL 构造和响应规范化。
- 单元测试 Rust 加密存储的读取、写入、迁移和解密失败行为。
- 使用模拟 `PlaybackInfo` 响应单元测试播放源选择。
- 在可行范围内使用 Mock 集成测试 Tauri commands，包括服务器校验、登录和播放会话创建。
- 至少使用一个包含电影、剧集、音乐、照片和直播电视的真实 Emby 服务器进行手动测试。
- 手动测试 mpv 缺失、mpv 启动失败、Direct Play 失败和转码回退路径。

## 实现备注

- 在依赖提交工作流前，项目需要初始化为 Git 仓库。
- 具体本地加密实现应在实现计划阶段选择。
- 第一轮播放实现应优先保证可观察性和清晰日志，而不是激进的自动回退。
- 如果前端请求导致 Token 暴露成为问题，则将全部 Emby API 调用迁移到 Rust commands 后方。
